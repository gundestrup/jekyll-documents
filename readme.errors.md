# Known Errors & Issues

Last reviewed: 2026-07-23 (verified against source + installed `jekyll-4.4.1` / `liquid-4.0.4`)
Last updated: 2026-07-23 (fixes applied — see status markers below)

## Bugs

### 1. `file_type_icon_tag` filter never receives Liquid context — `icon_set` config ignored

**Severity**: High — **✅ FIXED**

`file_type_icon_tag` at `lib/jekyll/documents/file_type_icons.rb:120` expects `context:` as a keyword argument. But in Liquid filter syntax (`{{ doc.file_type | file_type_icon_tag }}`), the context is **not** automatically passed. So `context` was always `nil`, meaning the configured `icon_set` was never read from site config — it always fell back to `"color"`.

**Fix applied**: The generator now bakes `icon_url` and `icon_set` into each document's `data` during generation (`lib/jekyll/documents/generator.rb:65-66`). All templates now use `{{ doc.icon_url }}` directly instead of the `file_type_icon_tag` filter:

- `_includes/latest_documents.html:6` — uses `{{ doc.icon_url | relative_url }}`
- `_includes/documents_list.html:5` — uses `{{ doc.icon_url | relative_url }}`
- `_layouts/document.html:16` — uses `{{ page.icon_url | relative_url }}`

A new class method `FileTypeIcons.icon_for(file_type, icon_set)` was added to allow the generator to resolve icon URLs at build time without needing a Liquid context.

**Related bug**: `Jekyll::Drops::SiteDrop#config` always returns `nil` in Liquid — this independently broke `site.config.documents.latest_default_count` fallback in `_includes/latest_documents.html`. The include now uses a simple `| default: 5` fallback (the `{% latest_documents %}` tag remains the recommended approach for config-aware rendering).

### 2. `rel_path` slicing assumes `site.source` prefix without trailing slash guard

**Severity**: Low — **✅ FIXED**

`lib/jekyll/documents/generator.rb:38` previously used:

```ruby
rel_path = path.start_with?(site.source) ? path[(site.source.length + 1)..] : path
```

**Fix applied**: Replaced with `String#delete_prefix`:

```ruby
rel_path = path.delete_prefix("#{site.source}/")
```

This is cleaner, handles edge cases (trailing slash, no match) gracefully, and removes the dead `else` branch.

---

### 3. `parse_filename` uses `Date.parse` instead of `Date.new`

**Severity**: Low — **✅ FIXED**

`lib/jekyll/documents/generator.rb:122-128` previously used `Date.parse` on a string assembled from regex captures.

**Fix applied**: Replaced with `Date.new(y, m, d)` using integer-converted captures:

```ruby
y = Regexp.last_match(1).to_i
m = Regexp.last_match(2).to_i
d = Regexp.last_match(3).to_i
date = Date.new(y, m, d)
```

The existing `rescue ArgumentError` still handles invalid dates (e.g. month 13). Existing tests confirm this behavior is unchanged.

## Best Practices Issues

### 4. Missing search JavaScript

**Severity**: Medium — **✅ FIXED**

`_includes/documents_search.html` previously provided only an `<input>` and `<div>` with no JavaScript.

**Fix applied**: The include now loads Lunr.js and the existing `assets/js/documents-search.js`:

```html
<script src="https://unpkg.com/lunr/lunr.js"></script>
<script src="{{ '/assets/js/documents-search.js' | relative_url }}"></script>
```

The search JS (`assets/js/documents-search.js`) was already present with full Lunr-based search, icon rendering, debounced input, and fallback simple-contains search. It fetches `/documents.json` (generated by `JsonIndexGenerator`), builds a Lunr index, and renders results with icons.

### 5. `category_list.html` references folder icon incorrectly

**Severity**: Low — **✅ FIXED**

`_includes/category_list.html:5` previously used `{{ site.documents.icon_set }}` which resolved to `nil` (since `site.documents` is an array, not config).

**Fix applied**: The template now reads `icon_set` from the first document's baked data:

```liquid
{% assign first_doc = site.documents | first %}
{% assign icon_set = first_doc.icon_set | default: 'color' %}
```

This works because the generator now bakes `icon_set` into every document (see Issue #1 fix).

### 6. No active `jekyll-sitemap` integration

**Severity**: Low — **✅ FIXED**

The gemspec description says "integrates with jekyll-sitemap" but there was no active integration code.

**Fix applied**: `TextStaticFile` (used for the JSON index) now exposes a `data` hash with `"sitemap" => false`:

```ruby
@data = { "sitemap" => false }
```

This tells `jekyll-sitemap` to exclude the generated `documents.json` from the sitemap (it's a search index, not a page).

### 7. Ruby 2.7 is EOL

**Severity**: Low — **✅ FIXED**

`required_ruby_version` was `">= 2.7"`. Ruby 2.7 reached EOL in March 2023, and Ruby 3.0 reached EOL in April 2024.

**Fix applied**: Bumped to `">= 3.1"` in `jekyll-documents.gemspec`. Also updated `.rubocop.yml` `TargetRubyVersion` from `2.7` to `3.1` to match.

### 8. `public_class_method :new` is a no-op

**Severity**: Trivial — **❌ Not a bug — keeping as-is**

`lib/jekyll/documents/tags/latest_documents.rb:11` — `public_class_method :new` was initially assumed to be a no-op since `new` is public by default.

**Finding during implementation**: Removing this line caused 29 test failures with `NoMethodError: private method 'new' called for LatestDocumentsTag:Class`. This is because `Liquid::Tag` (the parent class) makes `new` private. The `public_class_method :new` line is **required** to re-expose it. The original assessment was incorrect — this is not a no-op.

### 10. `Gemfile.lock` is gitignored but CI expects it

**Severity**: Low — **Confirmed, not yet fixed**

`.gitignore:6` excludes `Gemfile.lock`. For a gem this is conventional, but the CI workflow (`.github/workflows/test.yml:26`) uses `bundler-cache: true` which expects a lock file to exist for caching. This may cause CI cache misses on every run (not a correctness bug, just lost caching efficiency).

## Additional issues found during review (not previously documented)

### 11. Duplicate/competing "latest documents" implementations

**Severity**: Medium — **Partially mitigated**

There are two separate implementations:

- The Liquid **tag** `{% latest_documents %}` (`LatestDocumentsTag`) — correctly reads config server-side.
- The Liquid **include** `_includes/latest_documents.html` — was broken by `SiteDrop#config` nil-trap.

**Mitigation applied**: The include now uses `{{ doc.icon_url }}` (baked data) and a simple `| default: 5` count fallback. It no longer tries to read `site.config`. The `{% latest_documents %}` tag remains the recommended approach for full config-aware rendering.

### 12. Inconsistent filter signatures for context passing

**Severity**: Low — **Mitigated (no longer impactful)**

`file_type_icon(file_type, context = nil)` takes `context` positionally, while `file_type_icon_tag(file_type, css_class:, alt:, context:)` takes it as a keyword.

**Mitigation**: Templates no longer call these filters directly — they use `{{ doc.icon_url }}` baked data instead. The filters remain available for custom use cases but the inconsistency no longer affects any built-in template.
